## エグゼクティブサマリー

### 1. このサービスは何か

PARTCOPYは、**自社サイトの構造パターンを分析・再構成するサイト構築基盤**です。自社サイト・自社LP・自社コンテンツ資産をセクション単位で再編集し、高品質なページを短時間で再構成します。

ユーザーは自社サイトのURLを入力するだけで、ヒーロー、料金表、FAQ、CTAなどの構成パターンを自動分析。気に入ったパーツを選んでCanvasに配置し、AIが色・フォント・余白を統一した完成ページをHTMLとして出力します。参考サイトのURLも入力可能で、構造パターンの参考情報として活用できます。

### 2. 現在どこまでできているか

- 18サイト（Stripe, Salesforce, Figma, Notion 等）から**283セクション**を抽出済み
- 14カテゴリへの自動分類、平均信頼度 **78.5%**
- Canvas上でのパーツ配置・並べ替え・プレビュー・HTMLエクスポートが動作
- Claude APIによるCSS統一（AI最適化）がβ版で稼働

### 3. 現在の最大課題

- **normalizing工程のハング:** 一部サイトでセクション分割処理がタイムアウトなしで停止する
- **CSS競合:** 異なるサイト由来のパーツを1つのHTMLに統合する際、クラス名衝突が発生する
- **法的整理:** 構造パターンの再利用に関する方針は定めたが、法務確認は未実施

### 4. 経営判断としてどう見るべきか

技術的にはMVP成立ラインに到達しつつある。ただし、**汎用サービス化には時期尚早**。直近は「SaaS・コーポレートLPのパーツ抽出」に対象を絞り、抽出精度とエクスポート品質を上げるべき。

## 技術詳細（9つの質問への回答）

### Q1. パイプラインのどこで落ちるか

```
URL入力 → [1.Download] → [2.Store] → [3.Section Detection] → [4.Classify+Store] → [5.Complete]
```

| 段階 | 処理内容 | 成功率 | リスク |
|------|---------|--------|--------|
| 1. Download | Puppeteerでページ取得＋アセット保存 | 95%（18/19） | bot対策で拒否 |
| 2. Store | HTML/CSS/画像をStorageに保存 | 100% | — |
| 3. Section Detection | DOM解析→セクション分割 | 完了すれば100% | **タイムアウト未設定でハング** |
| 4. Classify + Store | 分類＋DOM snapshot保存 | 100% | parseSectionDOMにタイムアウト未設定 |
| 5. Complete | ステータス更新 | 100% | — |

**対策（直近で実装）:** Step 3・4に30秒タイムアウトを追加。超過時はスキップして次へ進む。DOMサイズ上限も設定する。

### Q2. 「正規化（normalizing）」の処理内容

「normalizing」ステータスは3つの処理をカバーする。

1. **セクション検出** — セマンティック要素（`<section>`, `<header>`等）の収集＋サイズ超過時の再帰分解
2. **分類** — CSSクラス名・ID・テキスト・DOM構造から14カテゴリへヒューリスティック分類（LLMなし）
3. **DOM凍結** — computed style付きでDOMツリーを保存。最もメモリ消費が大きくハングの主要因

### Q4. セクション分割のロジック

3段階で分割する。

1. **セマンティック要素を優先収集** — `<header>`, `<nav>`, `<section>`, `<article>`, `<aside>`, `<footer>`
2. **サイズ超過を再帰分解** — ビューポートの1.5倍を超えるセクションは子要素へ分解（深さ上限5）
3. **未カバー要素をフォールバック追加** — `<body>` の直接子要素で漏れた部分を拾う

最小閾値: 高さ40px × 幅200px。`<script>`, `<style>`, `<noscript>` 等は除外。

## 法的リスクと回避方針

### 何が危険か

- **他社サイトのテキスト・画像・ロゴ・商標をそのまま使うこと** — 明確に著作権・商標権侵害
- **他社サイトのHTMLコードをそのまま再配布すること** — 著作物の複製にあたる可能性

### どこまでならセーフ寄りか

- **コードの再利用ではなく、構造パターンの参考** — レイアウトそのものは「アイデア」であり著作権の保護対象外
- **HTMLタグの構成順序や余白・配色の設計パターン** — 機能的表現として保護されない領域
- 実際にWebデザイン業界では、優れたサイトの構造を参考にして自社サイトを作ることは標準的な慣行
- **画像・ロゴ・商標は参考パターンから完全に除外** — 参考サイトのブランド要素は一切出力に含まない

### 回避方針（プロダクトとしての対策）

1. **コードの再利用ではなく構造パターンの参考** — 参考サイトからはレイアウトパターンのみを抽出し、コードそのものは再利用しない
2. **画像・ロゴ・商標は除外** — 参考サイトの画像・ロゴ・商標・固有コピーはプレースホルダーに差し替え、出力に含めない
3. **最終出力は自社コンテンツ＋再生成CSSで構成** — 元サイトのブランド要素が残らない設計。自社サイトモードでは自社コンテンツをそのまま活用
4. **参考サイトの利用は構造パターンの参考情報として位置づけ** — 「3カラムの料金表」「左テキスト右画像のヒーロー」といったパターンの参考に限定

### 未対応事項

- 法務による正式な確認は未実施
- 利用規約の策定が必要
- 元サイトのブランド要素自動検出・警告機能は未実装

## アーキテクチャ概要

```
[ユーザー]
    │
    ├── URL入力
    │     ↓
    │   [Puppeteer] ─→ HTML/CSS/画像 ─→ [Supabase Storage]
    │     ↓
    │   [Section Detector] ─→ セクション分割（ルールベース）
    │     ↓
    │   [Classifier] ─→ 14カテゴリ分類（ヒューリスティック、LLMなし）
    │     ↓
    │   [Supabase DB] ← セクション情報保存
    │
    ├── ライブラリ ←── 横断検索・フィルタ
    │
    ├── Canvas ←── ドラッグ&ドロップで配置
    │
    ├── AI最適化（任意） ←── Claude API でCSS統一
    │
    └── HTMLエクスポート ─→ 完成ページをダウンロード
```

**技術スタック:** React + TypeScript / Express + Puppeteer / Supabase（DB・Storage） / Claude API（AI最適化のみ）

## 経営判断のための結論

### Go要素

- 抽出実績あり（18サイト・283セクション）
- サイト構造のパーツ化は技術的に成立し始めている
- AIコストが極めて低い（$0.05未満/ページ）
- 「自社サイトの構造パターンを分析・再構成し、参考サイトのパターンも活用できる」という差別化の方向性は明確
- 類似の完成度のプロダクトは国内にほぼない

### No-Go要素

- 汎用サイト対応はまだ早い（SPA・bot対策サイトで失敗する）
- 法務整理なしで対外訴求すると危険
- normalizeハングとCSS競合を放置したまま拡大すると品質が破綻する
- AI最適化の出力品質が「そのままデプロイできる」レベルに達していない

### 推奨判断

**今すぐフルプロダクト化ではなく、対象業種・対象サイト種別を絞った限定MVPとして継続開発。**

具体的には：
- **対象:** SaaS / BtoB コーポレートLP に限定
- **直近の打ち手:** normalizeタイムアウト実装済み、品質フィルタ実装済み、CSSスコープ化実装済み
- **判断マイルストーン:** 上記3点の解決後に、社内ユーザーでの実運用テストを実施し、プロダクト化の可否を再評価
